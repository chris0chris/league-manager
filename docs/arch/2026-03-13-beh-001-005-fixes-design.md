## BEH-001 & BEH-002 ‚Äî Accordion open by default + scroll-to-collapse

### Problem

`GamedayMetadataAccordion.tsx` uses an uncontrolled `<Accordion>` with no `defaultActiveKey`, so it starts closed on every load (BEH-001). The v3.1.7 scroll handler that collapsed the accordion when `scrollY > 50` was not migrated (BEH-002).

### Design

**`GamedayMetadataAccordion.tsx`**

- Convert `<Accordion>` to controlled: add internal `activeKey` state initialised to `"0"` (open).
- Add optional `forceCollapsed?: boolean` prop.
- A `useEffect` watches `forceCollapsed`: when it becomes `true`, set `activeKey` to `undefined`. **When it returns to `false`, do nothing** ‚Äî the accordion stays in its current state (collapsed). The user must manually re-open it by clicking the header. This matches v3.1.7 behaviour, where scroll-back-up did not auto-reopen.
- Pass `activeKey` and `onSelect` (which sets `activeKey` from user interaction) to `<Accordion>`.

**`ListDesignerApp.tsx`**

- Add `const [isMetadataCollapsed, setIsMetadataCollapsed] = useState(false)`.
- A `useEffect` attaches a `window` scroll listener on mount:
  - The handler must be a named function reference (not an inline arrow) so the `removeEventListener` cleanup correctly removes it.
  - Register with `{ passive: true }` to avoid browser performance warnings.
  - `scrollY > 50` ‚Üí `setIsMetadataCollapsed(true)`
  - `scrollY ‚â§ 50` ‚Üí `setIsMetadataCollapsed(false)` (permits future scroll-down collapses but does not reopen the accordion)
  - Returns cleanup function calling `removeEventListener` with the same named reference.
- Pass `forceCollapsed={isMetadataCollapsed}` to `<GamedayMetadataAccordion>`.

**Acceptance criteria:**
- Accordion is open when the page first loads.
- Scrolling past 50px collapses the accordion.
- Scrolling back above 50px does NOT reopen the accordion.
- Clicking the accordion header toggles it open/closed regardless of scroll position.

**Files changed:**
- `gameday_designer/src/components/GamedayMetadataAccordion.tsx`
- `gameday_designer/src/components/ListDesignerApp.tsx`

## BEH-003 ‚Äî Publish validation modal re-integrated

### Problem

`PublishConfirmationModal` is fully implemented but unused. The `onPublish` handler in `ListDesignerApp` calls `gamedayApi.publish()` directly without showing the modal, skipping validation review.

### Design

**`ListDesignerApp.tsx`**

- Import `PublishConfirmationModal` from `./modals/PublishConfirmationModal`.
- The `validation` prop passed to the modal is `FlowValidationResult` (the type returned by `useDesignerController`) ‚Äî this matches `PublishConfirmationModal`'s expected prop type exactly; no casting needed.
- Add `const [showPublishModal, setShowPublishModal] = useState(false)`.
- Change `onPublish` prop on `<GamedayMetadataAccordion>` to `() => setShowPublishModal(true)`.
- Extract publish API call into `handleConfirmPublish`:
  - **Close the modal immediately** (`setShowPublishModal(false)`) before awaiting the API call, so the UI is not blocked during the network request.
  - Awaits `gamedayApi.publish(parseInt(id))`.
  - On success: shows success notification, calls `loadData()` (fire-and-forget).
  - On failure: shows danger notification.
- Add `<PublishConfirmationModal>` to JSX:
  - `show={showPublishModal}`
  - `onHide={() => setShowPublishModal(false)}`
  - `onConfirm={handleConfirmPublish}`
  - `validation={validation}`
  - `onHighlight={handleHighlightElement}`

**Acceptance criteria:**
- Clicking Publish opens the `PublishConfirmationModal` showing current validation errors/warnings.
- Confirming closes the modal immediately, then calls the API.
- Success/failure notifications fire as before.

**Files changed:**
- `gameday_designer/src/components/ListDesignerApp.tsx`

## BEH-004 ‚Äî Delete confirmation modal

### Problem

`onDelete` uses `window.confirm(t('ui:message.confirmDeleteGameday'))` with a missing i18n key, displaying a raw key string in the browser dialog.

### Design

**New file: `gameday_designer/src/components/modals/DeleteGamedayConfirmModal.tsx`**

- React-Bootstrap `Modal`, centered, `size="sm"`.
- Body: danger `Alert` containing the translated confirmation message with the gameday name interpolated inline (e.g. _"Are you sure you want to delete **Spieltag 1**? This cannot be undone."_).
- Footer: Cancel button (`variant="secondary"`) + Delete button (`variant="danger"`).
- Props: `show: boolean`, `onHide: () => void`, `onConfirm: () => void`, `gamedayName?: string`.
- Uses `ui:message.confirmDeleteGameday` (with `{{name}}` interpolation) and `ui:button.deleteGameday` (pre-existing key ‚Äî do **not** add it again).

**i18n files** ‚Äî add to `message` section only:

`en/ui.json`:
```json
"confirmDeleteGameday": "Are you sure you want to delete \"{{name}}\"? This action cannot be undone."
```

`de/ui.json`:
```json
"confirmDeleteGameday": "M√∂chten Sie \"{{name}}\" wirklich l√∂schen? Diese Aktion kann nicht r√ºckg√§ngig gemacht werden."
```

**`ListDesignerApp.tsx`**

- Import `DeleteGamedayConfirmModal`.
- Add `const [showDeleteModal, setShowDeleteModal] = useState(false)`.
- Replace `onDelete` inline handler with `() => setShowDeleteModal(true)`.
- Extract delete logic into `handleConfirmDelete`:
  - Close modal immediately (`setShowDeleteModal(false)`).
  - Awaits `gamedayApi.deleteGameday(parseInt(id))`.
  - On success: navigates to `/`.
  - On failure: shows danger notification using `addNotification` (consistent with other error handlers in the file).
- Add `<DeleteGamedayConfirmModal show={showDeleteModal} onHide={...} onConfirm={handleConfirmDelete} gamedayName={metadata?.name} />` to JSX.

**Acceptance criteria:**
- Clicking Delete opens the modal showing the gameday name.
- Cancelling dismisses the modal with no side-effects.
- Confirming closes the modal, calls the API, navigates to `/` on success, shows error notification on failure.
- No `window.confirm` calls remain.

**Files changed:**
- `gameday_designer/src/components/modals/DeleteGamedayConfirmModal.tsx` _(new)_
- `gameday_designer/src/components/ListDesignerApp.tsx`
- `gameday_designer/src/i18n/locales/en/ui.json`
- `gameday_designer/src/i18n/locales/de/ui.json`

## BEH-005 ‚Äî Auto-save debounce restored to 1500ms

### Problem

`ListDesignerApp.tsx` debounces auto-save at `2000ms` instead of the v3.1.7 value of `1500ms`.

### Design

- In the auto-save `useEffect`, change the `setTimeout` delay argument from `2000` to `1500`. Identify the call by its context: it is the `setTimeout` inside the `pendingSaveRef.current.timer = setTimeout(async () => { ... })` block within the `flowState` change `useEffect`.

**Acceptance criteria:**
- Auto-save fires 1500ms after the last state change (not 2000ms).

**Files changed:**
- `gameday_designer/src/components/ListDesignerApp.tsx`
